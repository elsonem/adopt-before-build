# Contributing to adopt-before-build

Thank you for considering a contribution. This catalog improves with real-world experience — every tool suggestion, correction, or adoption story makes it more useful for the next person.

## English

### What we accept

| Type | Description |
|---|---|
| ✅ New tool | A free/OSS tool for an existing corporate area |
| ✅ New area | A corporate department not yet covered |
| ✅ License correction | A license that is wrong or outdated |
| ✅ Adoption story | Real experience using a tool internally |
| ✅ Tool removal | A tool that is abandoned, relicensed, or no longer free |
| ❌ Paid tools | Even with a free tier — this catalog is OSS/self-hosted only |
| ❌ Tools requiring IT to operate daily | The user must be able to operate it, not just install it |

### Criteria for a new tool

Every tool must meet **all three**:

1. **Free for internal use** — license explicitly allows self-hosted internal deployment at no cost
2. **Actively maintained** — last release within 12 months, issues being addressed
3. **Non-technical user can operate it** — a business user configures and uses it without writing code

### How to contribute

1. **Fork** this repository
2. **Create a branch** with a descriptive name: `add-invoiceninja-finance` or `fix-moodle-license`
3. **Make your change** following the format below
4. **Open a Pull Request** with a short description of what you are adding and why

### Format for a new tool entry

When adding a tool to any catalog file, use this format:

```markdown
| Area | [Tool Name](https://tool-url.com) | LicenseType | One-line description of what a non-technical user can do with it |
```

**Good description example:**
> Track employee leave requests and approvals with an automatic calendar — no spreadsheet needed.

**Bad description example:**
> Open-source HRMS with RESTful API and Docker support.

The description should speak to the **business user**, not the developer.

### Opening an issue instead

Not comfortable with Pull Requests? Open an issue with:

- **Title:** `[SUGGEST] Tool name — Area`
- **Body:** tool name, URL, license, which area it fits, and why it meets the three criteria

## Português

### O que aceitamos

| Tipo | Descrição |
|---|---|
| ✅ Nova ferramenta | Uma ferramenta gratuita/OSS para uma área corporativa existente |
| ✅ Nova área | Um departamento corporativo ainda não coberto |
| ✅ Correção de licença | Uma licença errada ou desatualizada |
| ✅ Relato de adoção | Experiência real usando uma ferramenta internamente |
| ✅ Remoção de ferramenta | Uma ferramenta abandonada, relicenciada ou que não é mais gratuita |
| ❌ Ferramentas pagas | Mesmo com free tier — este catálogo é somente OSS/self-hosted |
| ❌ Ferramentas que exigem TI para operar diariamente | O usuário de negócio precisa conseguir operar, não apenas instalar |

### Critérios para uma nova ferramenta

Toda ferramenta precisa atender aos **três critérios**:

1. **Gratuita para uso interno** — a licença permite explicitamente deploy self-hosted interno sem custo
2. **Ativamente mantida** — último release nos últimos 12 meses, issues sendo respondidas
3. **Usuário não-técnico consegue operar** — um usuário de negócio configura e usa sem escrever código

### Como contribuir

1. **Fork** este repositório
2. **Crie uma branch** com nome descritivo: `add-invoiceninja-financeiro` ou `fix-moodle-licenca`
3. **Faça sua alteração** seguindo o formato abaixo
4. **Abra um Pull Request** com uma descrição curta do que está adicionando e por quê

### Formato para nova entrada de ferramenta

Ao adicionar uma ferramenta a qualquer arquivo do catálogo, use este formato:

```markdown
| Área | [Nome da Ferramenta](https://url-da-ferramenta.com) | TipoDeLicença | Descrição em uma linha do que um usuário não-técnico consegue fazer com ela |
```

**Exemplo de boa descrição:**
> Controle solicitações de férias e aprovações com calendário automático — sem planilha.

**Exemplo de descrição ruim:**
> HRMS open-source com API RESTful e suporte a Docker.

A descrição deve falar para o **usuário de negócio**, não para o desenvolvedor.

### Preferir abrir uma issue

Não se sente confortável com Pull Requests? Abra uma issue com:

- **Título:** `[SUGESTÃO] Nome da ferramenta — Área`
- **Corpo:** nome da ferramenta, URL, licença, qual área se encaixa e por que atende aos três critérios
